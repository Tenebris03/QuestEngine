Accessibility (a11y) Best Practices for HTML
Grundprinzipien (WCAG)
Perceivable – Inhalte müssen wahrnehmbar sein

Operable – UI muss bedienbar sein

Understandable – Inhalte müssen verständlich sein

Robust – Inhalte müssen mit Assistenztechnologien funktionieren

1. Semantic HTML
Native Elemente nutzen
Struktur: <header>, <nav>, <main>, <section>, <article>, <aside>, <footer>

Aktionen: immer <button> statt klickbare <div>/<span>

Navigation: immer <a> statt <button> für Links

Gut

html
<button type="button">Save</button>
<nav><a href="/home">Home</a></nav>
Schlecht

html
<div class="button">Save</div>
<button routerLink="/home">Home</button>
Heading-Hierarchie
Nur eine <h1> pro Seite

Keine Level überspringen

Headings nicht für Styling missbrauchen

2. Formulare
Labels
Jedes Input braucht ein <label>

for und id verbinden

Placeholder ≠ Label

Gruppen mit <fieldset> + <legend>

Input-Attribute
Passende type-Werte (email, number, date …)

required für Pflichtfelder

autocomplete für gängige Felder

Fehlerbehandlung
Fehlertexte mit aria-describedby verknüpfen

Ungültige Felder: aria-invalid="true"

Dynamische Fehler: role="alert"

3. Bilder & Medien
Alt-Texte
Jedes <img> braucht alt

Dekorativ: alt=""

Bedeutungsvolle Bilder: beschreibender Text

Keine Phrasen wie „Bild von …“

Icons
Bedeutungsvolle Icons: aria-label oder versteckter Text

Dekorativ: aria-hidden="true"

4. ARIA
Grundregeln
Erste Regel: Nutze ARIA nur, wenn HTML nicht reicht

ARIA ändert Semantik, nicht Verhalten

Immer mit Screenreader testen

Wichtige Attribute
aria-label / aria-labelledby – Name

aria-describedby – Zusatzinfos

aria-hidden="true" – für Deko

aria-live – dynamische Inhalte

Rollen
Nur wenn nötig: dialog, alert, navigation, search, banner, contentinfo

5. Tastaturbedienbarkeit
Fokus
Alle interaktiven Elemente müssen per Tastatur erreichbar sein

tabindex="0" nur wenn nötig

Nie positive Tabindex-Werte

tabindex="-1" für programmatischen Fokus

Fokus sichtbar lassen (:focus-visible)

Skip Links
Erster Fokuspunkt: „Skip to main content“

Keyboard Events
Enter & Space für custom Controls unterstützen

6. Links & Buttons
Wann was?
Link (<a>) → Navigation

Button (<button>) → Aktion

Link-Text
Aussagekräftig

Keine „hier klicken“

Bei generischem Text → aria-label

Buttons
Immer type setzen

Klarer Name (Text oder aria-label)

7. Tabellen
Datentabellen
<table> nur für echte Tabellen

Header: <th scope="col|row">

Beschreibung: <caption> oder aria-label

Komplexe Tabellen: headers + id

8. Farben & Kontrast
Nicht nur Farbe nutzen
Immer zusätzliche Hinweise (Text, Icons, Muster)

Kontrastwerte
Normaler Text: 4.5:1

Großer Text: 3:1

UI-Elemente: 3:1

9. Dynamische Inhalte
Loading States
aria-busy="true"

Spinner mit Textalternative

Modals
role="dialog" oder alertdialog

aria-modal="true"

Beschriftung via aria-labelledby

Fokus im Modal halten

Fokus nach Schließen zurückgeben

10. Testing Checklist
Tastaturbedienung testen

Screenreader testen (NVDA, VoiceOver, JAWS)

Automatisierte Tests (axe, Lighthouse, WAVE)

Kontrast prüfen

Zoom auf 200% testen

Fokusmanagement prüfen

Alt-Texte vorhanden

Formulare korrekt beschriftet

Interaktive Elemente erreichbar
